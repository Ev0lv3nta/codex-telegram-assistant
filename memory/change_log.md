# Журнал изменений памяти

- Реализованы `session lease`, пользовательский приоритет и первый heartbeat/autonomy runner.
- Добавлены owner-request'ы через `system/tasks/autonomy_requests.md` и короткий журнал автономности в `system/tasks/autonomy_journal/YYYY-MM-DD.md`.
- Автономный prompt переведён на workspace-driven чтение памяти вместо инжекта snapshot-дампа.
- Постепенно убраны лишние planner/heartbeat слои, а wake-up логика упрощена до одного реального автономного цикла.
- Исправлены важные баги вокруг `waiting_user`, scheduled follow-up, active chat по командам, дублирующих блокеров и лишнего операционного шума в owner-facing отчётах.
- Новый крупный шаг: автономность переведена на более state-driven модель с `mode`, `next_wakeup`, `active mission` и короткими автономными сеансами вместо тупого fixed-heartbeat polling.
- Inline continuation теперь задаётся через `[[autonomy-next]]` с `DELAY_SEC: 0`, а отложенное продолжение — через положительный `DELAY_SEC`.
- Idle-контур ужат: без пользы он спит, а не пытается произвести активность ради активности.
- Живой рестарт после этого уже выполнен; state-driven runtime подтверждён в `bot_state.db`.
- Зафиксировано языковое правило: по умолчанию отвечать владельцу по-русски, если он явно не попросил другой язык.
- Закрыт ложный хвост `waiting_user` в `autonomy_tasks`, чтобы старое ожидание не загрязняло runtime-состояние.
- Добавлен отдельный `wake_event` для автономности: новые пользовательские сообщения теперь могут будить автономный worker сразу, без ожидания частого страховочного poll.
- `AUTONOMY_LOOP_POLL_SEC` в коде и конфиге поднят с `60` до `1800` секунд; добавлены тесты на новый wake-up путь и на helper, который будит автономность по входящему сообщению.
- Добавлен self-restart helper и marker-state для `personal-assistant-bot.service`: теперь запрос на рестарт можно зафиксировать до обрыва процесса и после следующего старта увидеть, что рестарт уже был наблюдён.
- Уточнены инструкции на self-restart: промежуточные сообщения больше не должны звучать так, будто рестарт "ещё только будет сделан", если сервис может уже успеть перезапуститься до следующего ответа.
- Усилен self-restart helper: теперь он ждёт фактического наблюдения нового `MainPID` / `ActiveEnterTimestampMonotonic`, а не только записи request-marker, и только после этого считает рестарт подтверждённым.
- На старте бота после подтверждённого self-restart теперь ставится обычная системная задача в очередь, чтобы сообщение `рестарт прошёл успешно` проходило через стандартный агентный путь ответа, а не отдельной заглушкой.
- В активное owner-request-поручение по самоулучшению дописаны конкретные внешние референсы: `openclaw.ai`, `openclaw/openclaw`, `joi-lab/ouroboros`, `HKUDS/nanobot`.
- Формат результата по этому исследованию уточнён: итог нужен не в виде обычного ответа в чат, а как короткий Markdown-файл с сильными идеями для реализации у нас.
- Зафиксировано правило темпа для автономности: если уже идёт живая исследовательская линия без внешнего блокера, не растягивать её редкими follow-up'ами на десятки минут; сначала делать плотный короткий блок работы, а уже потом уходить в длинный сон.
- Follow-up модель в коде упрощена: продолжение активной линии теперь может переиспользовать ту же автономную задачу как следующий запланированный wake-up вместо создания отдельной pending-задачи; релевантные тесты `autonomy_store` и `autonomy_worker` проходят.
- Исправлен ещё один owner-facing хвост: компрессор автономных уведомлений больше не должен сжимать содержательные результаты до одной первой строки; теперь он может пропускать 2-3 осмысленные строки, а тест на этот кейс добавлен.
- Правило tightened ещё сильнее: для owner-facing автономных сообщений убран сам постгенерационный summary-компрессор; после генерации модели в чат должен уходить её собственный текст, а не технически ужатый пересказ.
- После этого найден и локально закрыт ещё один хвост: в owner-facing сообщение не должны протекать служебные control-блоки `[[autonomy-next]]`; теперь убираются именно они, без возврата к агрессивному урезанию нормального текста.
- Owner-request по внешним автономным референсам закрыт реальным deliverable: подготовлен и отправлен Markdown-файл с краткой сводкой сильных идей для самоулучшения ассистента; активный inbox по этой линии очищен.
- Следом открыт новый owner-request на полную итерационную реализацию согласованных улучшений из итогового Markdown; в первую итерацию добавлен отдельный `topics/assistant-constitution.md` и он подключён в prompt-контур как стабильное identity/constitution ядро.
- Дополнительно уточнена граница этой новой реализации: разрешение владельца относится именно к автономному контуру и его owner-facing поверхностям, а не к произвольному общему рефакторингу всего бота.
- В owner-facing слое автономного контура начата следующая практическая итерация: добавлен компактный `/pulse`, а затем поверх него — inline-кнопка обновления, чтобы владелец мог смотреть текущее состояние автономности без ручного повторного ввода команды.
- Следующий owner-facing хвост на `/pulse` тоже закрыт: формат следующего wake-up сделан человекочитаемым в MSK вместо сырого ISO-значения, чтобы pulse читался как пользовательская поверхность, а не как отладочный вывод.
- В prompt автономного heartbeat-контура добавлен лёгкий self-review слой для project/maintenance/review шагов: перед самоизменениями ассистент теперь должен коротко фиксировать для себя, что меняет, зачем, какой главный риск и как проверит результат.

- Зафиксировано новое рабочее разрешение от владельца: в рамках текущей реализации автономного компаньона необходимые self-restart'ы можно делать без отдельного запроса каждый раз, если они реально нужны для подхвата уже готовых изменений.
- Доработан owner-facing смысл `/pulse`: при отсутствии активной миссии он теперь может показать ближайший pending-шаг, чтобы автономность выглядела как понятный поток `что будет дальше`, а не как пустое состояние между шагами.

- Owner-facing поведение паузы в `/pulse` тоже уточнено: при `sleeping_idle` с выставленным idle-snooze pulse теперь должен явно писать `притушен до ...`, чтобы эффект кнопки был понятен без чтения внутренних mode-значений.
- Зафиксирован стратегический сдвиг следующего шага: минимальный owner-facing слой на базе `/pulse` уже достаточно собран, и дальше контур должен выходить из локальной полировки pulse к следующему слою из итогового Markdown — практической дисциплине self-review для самоизменений.
- Из живого использования найден и закрыт ещё один owner-facing баг: refresh-кнопка в `/pulse` больше не должна будить автономность и перезаписывать `next_wakeup`; пассивный просмотр состояния теперь отделён от активного пользовательского сигнала.
