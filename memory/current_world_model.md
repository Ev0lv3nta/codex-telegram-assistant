# Текущая картина мира

Обновлено: 2026-03-08 05:49 (MSK)

- Главный текущий проект уже доведён до состояния более автономного компаньона на базе state-driven контура, а не fixed-heartbeat worker'а.
- Базовая state-driven модель уже есть: `mode`, `next_wakeup`, `active mission`, короткие автономные сеансы и follow-up через `[[autonomy-next]]`.
- Пользовательский ввод остаётся приоритетнее автономности; idle-контур не должен шуметь без пользы.
- Активных owner-request-поручений по автономному компаньону больше нет: линия реализации по итоговому Markdown закрыта как выполненная.
- Foundation-слой уже поднят: отдельное identity/constitution ядро подключено в prompt-контур.
- Минимальный owner-facing слой уже собран через `/pulse`: текущее состояние, следующий шаг, refresh, пауза `6ч`, жёсткий stop/start автономности и вертикальный список кнопок без упора в ширину экрана.
- Практический self-review для автономных самоизменений уже существует как внутренний протокол, который не протекает в owner-facing сообщения.
- Стратификация активной памяти уже в основном собрана: `current_world_model`, `open_loops` и `initiative_backlog` снова разведены по ролям и не должны обратно слипаться в полуархив.
- Последний большой runtime-pass тоже уже закрыт: `active_mission` научена хранить собственную фазу (`running/scheduled/waiting_user`), так что внутренняя модель контура стала ближе к одному цельному потоку работы, а не к набору косвенных признаков.
- Следом закрыт и главный self-management хвост: в runtime появился счётчик продолжений одной линии и жёсткий guardrail против бесконечных мелких `[[autonomy-next]]`-цепочек; prompt теперь тоже прямо запрещает микродробление там, где задачу можно закрыть в текущем проходе.
- Финальная связка тоже уже пройдена: runtime cohesion, quiet-by-default, `/pulse` control layer, memory stratification и self-review собраны в один practically finished контур; дальше остаётся только обычная живая калибровка как эксплуатация, а не как незакрытый этап плана.
- В проекте теперь зафиксирован отдельный слой расписаний для точных и регулярных автономных задач: `autonomy_schedules` в `system/tasks/bot_state.db`, типы `once/daily/weekly`, timezone по умолчанию `Europe/Moscow`, owner-facing команды `/schedules`, `/schedule_pause`, `/schedule_resume`, `/schedule_delete`.
- Исторические детали реализации, закрытые этапы и накопленные технические хвосты должны жить в `change_log.md`, а не в этом файле.
